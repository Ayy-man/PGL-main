**Migration file** (`supabase/migrations/20260407_activity_log_table.sql`):

```sql
CREATE TABLE IF NOT EXISTS activity_log (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  action_type TEXT NOT NULL,
  target_type TEXT,
  target_id UUID,
  metadata JSONB,
  ip_address TEXT,
  user_agent TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_activity_log_tenant ON activity_log (tenant_id);
CREATE INDEX IF NOT EXISTS idx_activity_log_created ON activity_log (tenant_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_activity_log_user ON activity_log (user_id);
CREATE INDEX IF NOT EXISTS idx_activity_log_action ON activity_log (tenant_id, action_type);

ALTER TABLE activity_log ENABLE ROW LEVEL SECURITY;
-- NOTE: RLS policies configured in Supabase dashboard per project convention
```

Keep the authentication/authorization check using the session client (`createClient()`) for the auth check (lines 26-43 -- verifying user exists and has admin role). But switch the data query (line 111-134) to use admin client with explicit tenant_id filter:

1. Add import: `import { createAdminClient } from "@/lib/supabase/admin";`
2. After auth check succeeds and `tenantId` is extracted from `user.app_metadata?.tenant_id`, create admin client: `const adminSupabase = createAdminClient();`
3. Replace the query builder (line 111 `let query = supabase.from(...)`) to use `adminSupabase.from("activity_log")` instead of `supabase.from("activity_log")`
4. Add explicit tenant scoping to the query: `.eq("tenant_id", tenantId)` -- this is critical since admin client bypasses RLS, we must manually scope to tenant
5. Extract `tenantId` from user metadata right after the role check:
   ```typescript

     return NextResponse.json({ error: "No tenant associated" }, { status: 403 });

   ```

This approach is consistent with the project pattern: `logActivity()` already uses admin client for writes for the same reason (RLS bypass).

**Apollo search route** (`src/app/api/search/apollo/route.ts`):

Add `logActivity` call after a successful search response (after line 111, before the `return NextResponse.json(result, ...)` on line 113):

1. Add import at top: `import { logActivity } from "@/lib/activity-logger";`
2. After the search completes successfully and before returning, add a fire-and-forget log call:
   ```typescript
   // Fire-and-forget: log search activity
   logActivity({
     tenantId,
     userId: user.id,
     actionType: "search_executed",
     targetType: "persona",
     targetId: personaId ?? undefined,
     metadata: {
       resultCount: result.people.length,
       totalResults: result.pagination.totalResults,
       cached: result.cached,
       searchIdentifier,
       page,
     },
   }).catch(() => {}); // never block on logging
   ```
   Place this BEFORE the return statement. Do NOT await it -- fire and forget. The `.catch(() => {})` prevents unhandled rejection warnings (logActivity already catches internally, but this is defensive).
