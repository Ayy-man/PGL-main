Edit `src/app/[orgId]/layout.tsx`:

1. After the existing tenant query block (where `tenant = data;` is set) and BEFORE the `userName = ...` line, add a parallel pair of head-count queries scoped to the tenant. Pull the tenant id from `user.app_metadata?.tenant_id` (which is the established pattern in this codebase per Phase 2 decisions), falling back to `tenant.id` if missing.

```typescript
// Live counts for sidebar nav badges (RLS handles tenant scoping for the session client)
const tenantScopeId = (user.app_metadata?.tenant_id as string | undefined) ?? tenant.id;

const [personasCountResult, listsCountResult] = await Promise.all([
  supabase
    .from("personas")
    .select("*", { count: "exact", head: true })
    .eq("tenant_id", tenantScopeId),
  supabase
    .from("lists")
    .select("*", { count: "exact", head: true })
    .eq("tenant_id", tenantScopeId),
]);

const savedSearchCount = personasCountResult.count ?? 0;
const listsCount = listsCountResult.count ?? 0;
```

2. Both queries are intentionally inside the existing `try` block so any RLS/network failure surfaces in the same error path. Do not add a separate try/catch — defaulting to `?? 0` is enough so a single failed query never blocks the layout render.

3. Pass the new values into `<Sidebar … />` JSX:
```tsx
<Sidebar
  orgId={orgId}
  tenantName={tenant.name}
  logoUrl={tenant.logo_url}
  userRole={userRole}
  userName={userName}
  userInitials={userInitials}
  savedSearchCount={savedSearchCount}
  listsCount={listsCount}
/>
```

4. Do not change any other layout behavior — theme injection, MobileBottomNav, TopBar, main wrapper all stay identical.

Why: Server-side queries via the session client get RLS scoping for free. `Promise.all` keeps both queries parallel so the layout doesn't pay sequential cost. `head: true` means we get the count without dragging row data, which is the right pattern for nav badges.
