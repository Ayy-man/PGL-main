# Plan 34-01: Fix activity_log CHECK Constraint

## Goal

Fix the CHECK constraint on `activity_log` and all monthly partition tables that only allows 11 of 24 action types. The 13 newer types (tenant_created, user_invited, user_invite_accepted, tenant_confirmed, tenant_renamed, tenant_settings_updated, metrics_aggregated, profile_edited, tag_added, tag_removed, photo_uploaded, lead_owner_assigned, issue_reported) silently fail on insert because `logActivity()` catches errors and returns null.

## Tasks

### Task 1: Write migration to drop CHECK constraints
- Create `supabase/migrations/20260410_fix_activity_log_check.sql`
- DROP CONSTRAINT on `activity_log` parent table
- DROP CONSTRAINT on all partition tables: `activity_log_2026_02` through `activity_log_2026_06`
- No replacement constraint — validate in application code via the `ActionType` union type in `src/lib/activity-logger.ts`

### Task 2: Push migration to Supabase
- Run `supabase db push` or apply via Supabase dashboard SQL editor
- If local CLI unavailable, provide the SQL for manual execution

### Task 3: Verify fix
- Confirm that inserting a row with `action_type = 'tenant_created'` succeeds
- Confirm that inserting a row with `action_type = 'user_invited'` succeeds
- Check that existing rows with the original 11 types are unaffected
