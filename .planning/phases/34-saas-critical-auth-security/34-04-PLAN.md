# Plan 34-04: Deactivated Tenant Gate

## Goal

When a super admin deactivates a tenant (`is_active = false`), enforce that at the middleware level so users of that tenant see a friendly "suspended" page and cannot access API routes. Currently only `[orgId]/layout.tsx` checks this, and middleware does not.

## Tasks

### Task 1: Add tenant active check to middleware
- Edit `src/middleware.ts`
- After resolving tenant context for tenant-scoped routes (`/[orgId]/*`), query the tenant's `is_active` status
- If `is_active = false`, redirect to `/suspended`
- Cache the check briefly (avoid per-request DB query) or piggyback on existing tenant resolution logic

### Task 2: Create suspended page
- Create `src/app/suspended/page.tsx`
- Simple centered page with message: "Your organization's account has been suspended"
- Include "Contact support" guidance and a "Sign out" button
- Match auth page styling (dark theme, centered card)
- Add `/suspended` to PUBLIC_ROUTES in middleware

### Task 3: Guard API routes
- Edit `src/lib/auth/session.ts` — in `requireTenantUser()` (or equivalent auth helper), add a check that the tenant is active
- Return 403 with message "Tenant suspended" if `is_active = false`
- This catches direct API calls that bypass middleware

## Verification
- [ ] Super admin deactivates tenant -> tenant user redirected to /suspended on next navigation
- [ ] /suspended page displays with sign-out button
- [ ] API calls from suspended tenant return 403
- [ ] Reactivating tenant restores access immediately
