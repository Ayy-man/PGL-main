<threat_model>
## Trust Boundaries

**Critical boundary (unchanged):** Client browser → Server API endpoints + Server Actions. Role-based access control enforced server-side in existing route handlers via `rbac.ts` → `requireRole()` / `checkPermission()`. This plan adds a client-side UX gate ON TOP of server authz — it does NOT replace it.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-03-01 | Elevation of Privilege | Client-side canEdit bypass | mitigate | Client gate is UX only. Server-side `rbac.ts` + per-route role checks remain authoritative. Task 7 manual verification step 6 explicitly tests that bypassing the disabled button (via direct API call) still returns 403. If any endpoint returns 200 to an assistant write, that's a pre-existing server-side bug NOT introduced by this plan — flag and fix separately. |
| T-39-03-02 | Spoofing | Role claim tampering | accept | Role is read from `user.app_metadata` which is immutable by client (set server-side only). The `canEdit` derivation happens in the server component (`page.tsx`) using `getCurrentUser()` which calls `supabase.auth.getUser()` with JWT validation. Client cannot forge a higher role. |
| T-39-03-03 | Information Disclosure | Disabled-but-visible write buttons reveal feature existence | accept | Assistants can already see the UI (they have read access). Disabled buttons with explanatory tooltips improve UX vs invisible features (they understand why they can't act). No new data exposed. |
| T-39-03-04 | Repudiation | Assistant attempts write, gets blocked | accept | Server-side audit logs (where they exist) already capture denied writes if hit. Client gate reduces the noise; server still authoritative for audit trail. |

**Key security assertion:** This plan does NOT change server behavior. If an assistant bypasses the disabled UI (e.g. by manually calling the API endpoint), they still get 403 from the existing server-side role check. Task 7 step 6 verifies this for at least one write endpoint; if it fails, that's a pre-existing vulnerability that must be fixed separately — do NOT rely on client disabled-state as the sole gate.
</threat_model>
