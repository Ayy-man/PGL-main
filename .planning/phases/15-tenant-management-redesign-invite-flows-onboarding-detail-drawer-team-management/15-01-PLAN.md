# Plan 15-01: Activity Logger Updates + New Action Types + Middleware Onboarding Redirect

## Goal
Extend the activity logging system with tenant lifecycle events and add middleware logic to detect new tenant admins who need onboarding.

## Tasks

### Task 1: Add new action types to activity logger
**File:** `src/lib/activity-logger.ts`

Add 6 new action types to the `ACTION_TYPES` array:
- `tenant_created` — when super admin creates a tenant
- `tenant_renamed` — when tenant name/slug is changed (store before/after in metadata)
- `tenant_settings_updated` — when tenant colors/logo are changed (store before/after)
- `tenant_confirmed` — when invited admin confirms tenant settings during onboarding
- `user_invited` — when any admin invites a user via email
- `user_invite_accepted` — when invited user completes onboarding

Update the `ActionType` type to include these new values.

### Task 2: Add onboarding redirect to middleware
**File:** `src/middleware.ts`

After session validation, check `user.app_metadata?.onboarding_completed === false`:
- If true and path is NOT `/onboarding/confirm-tenant` or `/api/auth/callback`, redirect to `/onboarding/confirm-tenant`
- Add `/onboarding/confirm-tenant` to the list of routes that require authentication but skip tenant resolution
- Ensure super_admin users are never redirected to onboarding (they don't need it)

### Task 3: Update auth callback for onboarding detection
**File:** `src/app/api/auth/callback/route.ts`

After exchanging the code for a session:
- Check if `user.app_metadata?.onboarding_completed === false`
- If so, redirect to `/onboarding/confirm-tenant` instead of the normal role-based redirect
- This catches the magic link click from the invite email

## Files Modified

- New action types are exported and usable

- Auth callback detects onboarding flag and redirects correctly
- Existing routes and flows are unaffected
