  <how-to-verify>
    Run these 7 scenarios in order. Check each off or note a failure.

    **Setup:**
    - Local dev server running (`pnpm dev`) or deployed to `pgl-main.vercel.app`
    - Logged in as super_admin in one browser (for admin UI)
    - Logged in as a tenant user in a second browser (or incognito) for report creation
    - Supabase SQL editor open for DB verification

    **Scenario 1: Reported event on new report creation**
    1. As tenant user, navigate to any prospect profile.
    2. Click "Report an issue" → select "Incorrect data" → enter description → submit (screenshot checkbox ON).
    3. Expect toast "Thanks — we'll take a look".
    4. In Supabase SQL editor: `SELECT event_type, actor_role FROM issue_report_events WHERE report_id = (SELECT id FROM issue_reports ORDER BY created_at DESC LIMIT 1);`
    5. Expected: exactly one row with `event_type = 'reported'`, `actor_role = 'tenant'`.
    ✅ PASS if matches / ❌ FAIL otherwise.

    **Scenario 2: status_changed event on status transition**
    1. As super_admin, open `/admin/reports/{id}` for the report from Scenario 1.
    2. Change status dropdown from `open` → `investigating`. Leave notes empty.
    3. Click Save. Expect success toast "Changes saved successfully".
    4. Refresh page. Timeline should show a new `{admin} changed status: Open → Investigating` row with no note body.
    5. SQL: `SELECT event_type, from_status, to_status FROM issue_report_events WHERE report_id = '{id}' ORDER BY created_at DESC LIMIT 1;`
    6. Expected: `event_type='status_changed'`, `from_status='open'`, `to_status='investigating'`.
    ✅ PASS if matches / ❌ FAIL otherwise.

    **Scenario 3: note_added event on notes-only change**
    1. Same report, edit admin notes to "Looking into this" and Save WITHOUT changing status.
    2. Refresh. Timeline should show `{admin} added a note` with the note body visible below.
    3. SQL: `SELECT event_type, note FROM issue_report_events WHERE report_id = '{id}' ORDER BY created_at DESC LIMIT 1;`
    4. Expected: `event_type='note_added'`, `note='Looking into this'`.
    ✅ PASS if matches / ❌ FAIL otherwise.

    **Scenario 4: Close-requires-note — UI + server both block**
    1. On same report, change status dropdown to `resolved` but CLEAR admin notes (empty textarea).
    2. Confirm the Save button is DISABLED and red helper text reads "A note is required when closing an issue."
    3. Open browser devtools → Network tab.
    4. In the console, attempt to bypass client-side: `fetch('/api/admin/reports/{id}', { method: 'PATCH', headers: {'Content-Type':'application/json'}, body: JSON.stringify({status:'resolved', admin_notes:''}) }).then(r => r.json().then(b => console.log(r.status, b)))`
    5. Expected: HTTP 400, `{error: "A note is required when closing an issue"}`.
    ✅ PASS if both client AND server block / ❌ FAIL otherwise.

    **Scenario 5: viewed_by_admin — 24h dedup**
    1. Log out super_admin. Log back in.
    2. Visit `/admin/reports/{id}`. Wait for load.
    3. Refresh the page.
    4. SQL: `SELECT COUNT(*) FROM issue_report_events WHERE report_id = '{id}' AND actor_user_id = '{super_admin_user_id}' AND event_type = 'viewed_by_admin' AND created_at > now() - interval '1 hour';`
    5. Expected: exactly 1 (single row despite two page loads within minutes).
    ✅ PASS if count == 1 / ❌ FAIL otherwise.

    **Sub-scenario 5b: viewed_by_admin — rapid-repeat dedup**
    1. As super_admin, pick a report you have NEVER viewed (or first clear rows: `DELETE is NOT allowed on audit log — instead use a report you have never opened, or a freshly-created one from Scenario 1`).
    2. Visit `/admin/reports/{id}`.
    3. Within a few seconds, refresh the page 2–3 more times.
    4. SQL: `SELECT COUNT(*) AS n FROM issue_report_events WHERE report_id = '{id}' AND actor_user_id = '{super_admin_user_id}' AND event_type = 'viewed_by_admin';`
    5. Expected: `n = 1`. A count > 1 indicates the 24h dedup window is not firing on the server loader.
    6. Note: an extremely fast double-load (same tick) is an accepted race per Threat T-38-10 — duplicate view events are noise. If you see `n = 2` only under that specific race and `n = 1` on sequential refreshes, still count as PASS with a note.
    ✅ PASS if `n = 1` on sequential refreshes / ❌ FAIL if n > 1 consistently.

    **Scenario 6: Backfilled events for existing reports**
    1. Pick an issue_report created BEFORE this phase was deployed (or just before Plan 01 ran). If no such row exists, create one pre-migration replay isn't feasible — instead confirm via SQL that backfilled rows have `actor_role='tenant'` and `event_type='reported'` with `created_at` matching the report's original `created_at`.
    2. SQL: `SELECT e.created_at AS event_time, r.created_at AS report_time, e.event_type, e.actor_role FROM issue_report_events e JOIN issue_reports r ON r.id = e.report_id WHERE e.event_type = 'reported' ORDER BY r.created_at LIMIT 5;`
    3. Expected: `event_time = report_time` for every row.
    4. Open the oldest pre-existing report in the admin UI. Confirm timeline shows at least a `Reported by {tenant}` row.
    ✅ PASS if backfill visible / ❌ FAIL otherwise.

    **Scenario 7: Responsive layout**
    1. On `/admin/reports/{id}` at viewport width ≥1024px: Admin Actions is a wider left column, TimelineCard is a narrower right column, side-by-side.
    2. Resize to 768px: TimelineCard stacks below Admin Actions.
    3. Confirm all sections ABOVE Admin Actions (header, submitter, description, snapshot, screenshot, URL, context) remain full-width at both breakpoints.
    ✅ PASS if layout matches / ❌ FAIL otherwise.

    **Optional Scenario 8: screenshot_expired (nice-to-have, not blocking)**
    - Hard to trigger manually because signed URLs are 60 min and require time passage.
    - Skip in UAT unless you can force a failure (e.g., delete the storage object and reload).

    **Report back:** List each scenario with PASS/FAIL. For any FAIL, include the unexpected output vs expected.
