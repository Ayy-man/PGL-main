### Pattern A — Server Action guard (Plans 02, 03)
Place immediately AFTER the existing auth/tenant resolution in each action,
BEFORE any call into `@/lib/lists/queries` or `@/lib/personas/queries`.

```typescript
import { requireRole } from "@/lib/auth/rbac";

// ...inside each exported async function action(...):
await requireRole("agent");
// Throws NEXT_REDIRECT for assistant — Server Actions surface this
// to the client as a navigation to "/{tenantId}" (existing rbac.ts
// behavior, lines 16-18). For the three lists actions that already
// wrap the body in try/catch and return { success: false, error },
// the redirect error should be allowed to propagate — do NOT
// swallow it into a { success: false } envelope, because the client
// canEdit UI prevents ever hitting this path under normal conditions
// and we want the redirect to fire if someone truly bypasses the UI.
```

### Pattern B — Route Handler guard (Plan 04)
Applies to exactly these 2 Route Handlers (both owned by Plan 04):
- `POST` in `src/app/api/apollo/bulk-enrich/route.ts`
- `PATCH` in `src/app/api/prospects/[prospectId]/notes/route.ts`

Place immediately AFTER the existing `if (authError || !user)` 401 block and
AFTER the `tenantId` extraction, BEFORE any mutation or external API call.

```typescript
import { hasMinRole } from "@/types/auth";
import type { UserRole } from "@/types/auth";

// ...inside the route handler, after the tenantId 401 guard:
const role = (user.app_metadata?.role as UserRole) || "assistant";
if (!hasMinRole(role, "agent")) {
  return NextResponse.json(
    { error: "Forbidden", message: "Your role does not permit this action" },
    { status: 403 }
  );
}
```

**Why not `await requireRole('agent')` in Route Handlers?**
`requireRole` calls Next.js `redirect()` (rbac.ts:13, 17, 19) which throws
`NEXT_REDIRECT`. In Server Actions that error is caught by the framework and
converted to a client-side navigation; in Route Handlers it escapes to the
generic `catch` block and surfaces as HTTP 500 — which violates the locked
`{ status: 403, error: 'Forbidden' }` contract in 42-CONTEXT.md §"Error
response shape". Every existing admin Route Handler in this codebase uses
the inline `user.app_metadata?.role` check for exactly this reason
(see `src/app/api/admin/tenants/route.ts:14-16` — ~20 files follow this
pattern). We mirror that pattern using `hasMinRole` so the hierarchy check
stays correct (agent/tenant_admin/super_admin all pass).

### Pattern C — Locked 403 JSON body (Route Handlers only)
```json
{ "error": "Forbidden", "message": "Your role does not permit this action" }
```
Status code: **403**. Do not omit the `message` field. Do not use `FORBIDDEN`
all-caps (the ROADMAP entry used that but CONTEXT.md §"Error response shape"
locks it to title-case `Forbidden`).

### 401 preservation
The existing `if (authError || !user)` → `{ error: 'Unauthorized' }`, 401
block stays untouched in every Route Handler. 401 = no session; 403 = session
exists, role insufficient. Tests MUST cover both.
