# Phase 23: Intelligence Dossier & Wealth Signal Timeline — Research

**Researched:** 2026-03-27
**Domain:** Enrichment pipeline extension, LLM dossier generation, timeline UI, Supabase schema
**Confidence:** HIGH

## Summary

Phase 23 splits the existing "Wealth Signals & Intelligence" `<WealthSignals>` component into two distinct sections: (1) an Intelligence Dossier — a structured, prose-rendered static brief generated by Claude/OpenRouter from all enrichment data at the end of the `enrich-prospect` Inngest workflow — and (2) a Wealth Signals Timeline — a live, per-user event feed backed by a new `prospect_signals` table, displayed as a vertical timeline with category badges and "NEW" indicators tracked via a `signal_views` table.

The existing enrichment pipeline (`src/inngest/functions/enrich-prospect.ts`) already does all the heavy lifting: ContactOut, Exa, SEC EDGAR, Market Data, and a Claude AI summary step. The dossier is a new Step 5.5 (after the existing claude summary step, before finalize) that sends ALL enrichment data in a richer prompt and saves a structured JSON to a new `intelligence_dossier` JSONB column on `prospects`. The Exa digest (`exa-digest.ts`) already outputs categorized signals with `source_url`; converting those to `prospect_signals` rows requires a DB write replacing the current `web_data` JSONB blob (or in addition to it for backward compat). SEC transactions in `insider_data.transactions` similarly become signal rows.

The profile page layout (`profile-view.tsx`) uses a 3-column CSS grid (3/6/3). The center column currently holds `<WealthSignals>` then `<MarketIntelligenceCard>` then Company Context. In Phase 23 the center column order becomes: Intelligence Dossier (new component) → Wealth Signals Timeline (new component, replaces `<WealthSignals>`) → Market Intelligence Card → Company Context.

**Primary recommendation:** Add the dossier generation as a new Inngest step at the end of `enrich-prospect`, write signals to the `prospect_signals` table during the same workflow, and build two new React components that consume the new DB columns/table instead of patching the existing `<WealthSignals>` component in-place.

### Pattern 1: New Inngest Step (Dossier Generation)
**What:** Add `step.run("generate-dossier", ...)` after the existing `generate-summary` step in `enrich-prospect`. Call `generateIntelligenceDossier()` which hits OpenRouter with a richer prompt. Save result to `prospects.intelligence_dossier` + `prospects.dossier_generated_at` + `prospects.dossier_model`.
**When to use:** Runs during every full enrichment. The dossier replaces/supplements `ai_summary`.
**Example (follows existing claude.ts pattern):**
```typescript
// In enrich-prospect.ts, after Step 5 (generate-summary):
const dossierResult = await step.run("generate-dossier", async () => {
  const dossier = await generateIntelligenceDossier({
    name, title, company,
    contactData: contactData.found ? { personalEmail: contactData.personalEmail, phone: contactData.phone } : null,
    webSignals: exaData.signals,
    insiderTransactions: edgarData.transactions,
    stockSnapshot: marketData.found ? { ticker: effectiveTicker } : null,
  });

  if (dossier) {
    await supabase.from("prospects").update({
      intelligence_dossier: dossier,
      dossier_generated_at: new Date().toISOString(),
      dossier_model: "anthropic/claude-3.5-haiku",
    }).eq("id", prospectId);
  }

  await updateSourceStatus(prospectId, "dossier", {
    status: dossier ? "complete" : "failed",
    at: new Date().toISOString(),
  });

  return { dossier, status: dossier ? "complete" : "failed" };
});
```

### Pattern 2: Signal Writes in Inngest (Prospect Signals)
**What:** After Exa digest and SEC EDGAR steps, write to `prospect_signals` table instead of (or in addition to) updating JSONB blobs. Each `DigestedSignal` becomes one row. Each SEC transaction becomes one row with `category: "sec_filing"`.
**When to use:** Inside the existing `fetch-exa` and `fetch-edgar` steps, after the existing `supabase.from("prospects").update({ web_data: ... })` call.
**Example:**
```typescript
// After web_data update in fetch-exa step:
if (digestedSignals.length > 0) {
  const signalRows = digestedSignals.map(s => ({
    prospect_id: prospectId,
    tenant_id: tenantId,
    category: s.category,
    headline: s.headline,
    summary: s.summary,
    source_url: s.source_url,
    event_date: null,   // Exa results don't have reliable dates
    raw_source: "exa",
  }));
  // Upsert to avoid duplicates on re-enrichment
  await supabase.from("prospect_signals").upsert(signalRows, {
    onConflict: "prospect_id,source_url",  // or a hash-based dedup key
    ignoreDuplicates: true,
  });
}
```

### Pattern 3: Intelligence Dossier Component (Prose Rendering)
**What:** New `<IntelligenceDossier>` client component. Receives `dossier: IntelligenceDossierData | null` prop. If null, shows empty state. If populated, renders structured prose sections (summary, career_narrative, wealth_assessment, company_context, outreach_hooks, quick_facts).
**When to use:** Top of center column in `profile-view.tsx`, replacing the old `<WealthSignals>` header section.
**Pattern:** Each section rendered as a labeled prose block with gold left-border accent (matching existing `ai_summary` treatment per STATE.md Phase 05-06 note: `border-l-2 borderColor=var(--border-gold)`).

### Pattern 4: Wealth Signals Timeline Component
**What:** New `<WealthSignalTimeline>` client component. Fetches signals from `/api/prospects/[id]/signals`. Renders vertical timeline — left dot connector rail, category badge pill (same icons as current `getCategoryIcon()`), headline, summary, source link, timestamp. "NEW" badge on unseen signals.
**When to use:** Center column below `<IntelligenceDossier>`, replacing the existing `<WealthSignals>` card grid.
**Seen-tracking pattern:** On mount, call `POST /api/prospects/[id]/signals/mark-seen` with `{ signalIds: string[] }`. Server upserts to `signal_views` table. On subsequent page load, signals already in `signal_views` for this user render without the "NEW" badge.

### Anti-Patterns to Avoid
- **Patching WealthSignals in-place:** The existing component mixes Exa cards and SEC table into one `<div>`. Phase 23 separates concerns into dossier + timeline. Don't try to adapt the existing component — replace it with two new ones.
- **Fetching signals client-side on every render without caching:** Use the server component page.tsx to fetch the initial batch of signals and pass them as props, with a client-side "load more" if needed.
- **Blocking enrichment on dossier failure:** Same pattern as existing steps — `try/catch` with graceful degradation. A failed dossier must not fail the whole enrichment workflow.
- **Re-inserting duplicate signals on re-enrichment:** Use `prospect_id + source_url` composite unique index (for Exa signals) and `prospect_id + filing_date + transaction_type + shares` for SEC signals to make upserts idempotent.
- **Storing `raw_text` in prospect_signals:** `raw_text` is debug-only per `DigestedSignal` type. Don't put it in the new table.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|

| Category icons | Custom icon set | Existing `getCategoryIcon()` from `wealth-signals.tsx` | Extract to shared util or duplicate into new component |

**Key insight:** The `chatCompletion()` function takes `system`, `userMessage`, `maxTokens`, and optional `model` — the dossier call just needs a richer prompt and higher `maxTokens` (~1500-2000 for structured JSON output). No new AI infrastructure needed.

## Common Pitfalls

### Pitfall 1: Intelligence Dossier JSON Parse Failure
**What goes wrong:** Claude returns JSON wrapped in markdown code fences (` ```json ... ``` `). If `JSON.parse` is called on the raw response, it throws.
**Why it happens:** LLMs frequently add code fence formatting even when told not to.
**How to avoid:** Use the same fence-stripping logic already in `exa-digest.ts` lines 81-84 before parsing.
**Warning signs:** `SyntaxError: Unexpected token` in Inngest logs.

### Pitfall 2: Duplicate Signals on Re-enrichment
**What goes wrong:** Every re-enrichment (>7 days stale) re-runs all Inngest steps. Without deduplication, `prospect_signals` accumulates duplicate rows for the same Exa URLs and SEC transactions.
**Why it happens:** The Exa and EDGAR steps run unconditionally on each enrichment cycle.
**How to avoid:** Create a composite unique index: `(prospect_id, source_url)` for Exa signals; `(prospect_id, raw_source, event_date, headline_hash)` or `(prospect_id, raw_source, event_date, transaction_type, shares)` for SEC signals. Use `.upsert(..., { ignoreDuplicates: true })`.
**Warning signs:** Signal count doubles with each enrichment; timeline shows identical entries.

### Pitfall 3: `signal_views` RLS — User Cross-Tenant Leakage
**What goes wrong:** If `signal_views` is not properly RLS-scoped to `tenant_id`, one tenant's users could mark another tenant's signals as seen.
**Why it happens:** `signal_views` joins on `user_id` but signals belong to prospects which belong to tenants.
**How to avoid:** Add `tenant_id` column to `signal_views` and enforce RLS the same way as `prospect_tags`. RLS policies configured in Supabase dashboard (per project convention in migration comment).
**Warning signs:** A user sees signals as "read" that they never opened.

### Pitfall 4: Dossier Step Increases Inngest Run Duration
**What goes wrong:** Adding a new LLM step to the already ~5-step enrichment flow adds 3-8 seconds per enrichment. At 5 concurrent enrichments (current limit), this is acceptable, but if `maxTokens` is set too high (e.g., 4000+), response time spikes.
**Why it happens:** Claude Haiku is fast but structured JSON at high token counts slows responses.
**How to avoid:** Cap dossier `maxTokens` at 1500-2000. The 6-field JSON is achievable in that budget. Use `claude-3.5-haiku` (not Sonnet) for cost and latency.
**Warning signs:** Inngest timeouts (step.run has a 2-minute default timeout, so this won't fail, but will appear slow in the Automations dashboard).

### Pitfall 5: `web_data` vs `prospect_signals` Dual State
**What goes wrong:** `web_data` JSONB column is still read by the old `<WealthSignals>` component (if it's kept anywhere) and by `generateProspectSummary()` (Step 5 in enrich-prospect). If signals are migrated to `prospect_signals` table but `web_data` is set to null, the existing `generateProspectSummary` call breaks.
**Why it happens:** Two data consumers of the same enrichment data during the transition.
**How to avoid:** Keep `web_data` write in `fetch-exa` step intact (backward compat). Write to `prospect_signals` IN ADDITION to `web_data`. Only the UI components change — `<WealthSignals>` is replaced, but `generateProspectSummary` keeps reading `exaData.signals` from Inngest step memory, not from DB.
**Warning signs:** Step 5 (generate-summary) produces empty summaries after Phase 23.

### Pitfall 6: Per-User Seen Tracking Requires User Context in API Route
**What goes wrong:** `/api/prospects/[id]/signals/mark-seen` needs the authenticated user's ID to write to `signal_views`. If called without auth, it throws.
**Why it happens:** The `signal_views` table is user-scoped. The route handler must call `supabase.auth.getUser()` and use the session client (not admin client) for the `signal_views` write.
**How to avoid:** Follow the same pattern as `/api/prospects/[id]/tags/route.ts` — use session client, extract `user.id` from `supabase.auth.getUser()`.
**Warning signs:** 401 errors when marking signals as seen; "seen" state never persists.

### Pitfall 7: `dossier_model` Column — Avoid Hardcoding Model Name in SQL
**What goes wrong:** Model names like `anthropic/claude-3.5-haiku` change over time. If the default is baked into a DB constraint, it breaks when switching models.
**How to avoid:** `dossier_model text` with no constraint — application sets the value. Document it in code only.

## Open Questions

1. **Backfill for existing enriched prospects**
   - What we know: Existing prospects already have `web_data.signals[]` and `insider_data.transactions[]` in JSONB. They will not have `prospect_signals` rows until they are re-enriched.
   - What's unclear: Should the planner include a one-time backfill Inngest function that reads existing JSONB and inserts `prospect_signals` rows? Or is re-enrichment (triggered on next profile view after 7 days) sufficient?
   - Recommendation: Include an optional backfill Inngest function that can be triggered once manually via Inngest dashboard. Not time-critical for launch but valuable for tenants with existing enriched prospects.

2. **`event_date` extraction for Exa signals**
   - What we know: Exa API returns `publishedDate` on results; `enrichExa()` maps it to `publishDate` on mentions; `digestExaResults()` does not currently capture it in `DigestedSignal`.
   - What's unclear: Is `publishedDate` reliably populated in Exa API responses?
   - Recommendation: Pass through as optional field (`event_date?: string | null`); timeline sorts by `created_at` when `event_date` is null.

3. **"NEW" badge semantics — global vs per-user**
   - What we know: Phase description says "per-user seen tracking via `signal_views` table."
   - What's unclear: Should "NEW" mean "never seen by this user" (per-user) or "created in the last N days" (time-based)?
   - Recommendation: Use `signal_views` per-user tracking as specified. On first page load, all signals are NEW. Mark-seen runs on mount → subsequent loads show no NEW badges.

4. **Dossier section: does it replace the existing `ai_summary` display?**
   - What we know: `ai_summary` is currently displayed in the `<ProfileHeader>` component (left column) as a gold left-border text block.
   - What's unclear: Phase 23 spec says "top of center column" for the dossier. If dossier exists, should `ai_summary` in the left column be hidden?
   - Recommendation: Keep `ai_summary` in `<ProfileHeader>` as fallback when `intelligence_dossier` is null. When dossier exists, `ai_summary` can be hidden or condensed since the dossier provides richer coverage.
