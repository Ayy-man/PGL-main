## Summary

Phase 25 installs the official shadcn AI component suite (available at `shadcn.io/r/{name}.json`) and builds a streaming research panel embedded in the prospect profile center column. The technical core is the Vercel AI SDK v6 streaming pattern — `createUIMessageStream` + `writer.write()` for multi-phase custom data streaming — paired with OpenRouter as the LLM provider (already established in the project) via the `@openrouter/ai-sdk-provider` package. The shadcn AI components integrate natively with this SDK's `useChat` hook.

The profile view currently has NO tabs — it is a continuous three-column scroll layout. The Dossier/Research toggle is a NEW UI pattern being introduced. The center column currently renders `<IntelligenceDossier>` + `<SignalTimeline>` + market/company context as stacked cards. Phase 25 adds a two-tab pill that swaps the `<IntelligenceDossier>` block with a `<ResearchPanel>` while Wealth Signals remain visible in both tabs.

The data model adds three new Supabase tables (`research_sessions`, `research_messages`, `research_pins`) following the exact migration convention used by Phases 23 and 24: CREATE TABLE + CREATE INDEX + ALTER TABLE ENABLE ROW LEVEL SECURITY + dashboard-configured RLS note.

**Primary recommendation:** Use `@openrouter/ai-sdk-provider` + `createUIMessageStream` for streaming, not the legacy `StreamData` class. Install all 11 shadcn AI components first, restyle to design system, then build API routes and panel.

### Pattern 1: Multi-Phase Custom Data Streaming (AI SDK v6)
**What:** Stream distinct phases (reasoning, tool, card objects, sources) as typed data parts alongside text
**When to use:** Any API route that needs to send structured non-text data to `useChat` on the client

```typescript
// Source: https://ai-sdk.dev/docs/ai-sdk-ui/streaming-data (AI SDK v6)
// app/api/prospects/[prospectId]/research/route.ts

import { createUIMessageStream, createUIMessageStreamResponse, streamText } from 'ai';
import { createOpenRouter } from '@openrouter/ai-sdk-provider';

const openrouter = createOpenRouter({
  apiKey: process.env.OPENROUTER_API_KEY!,
});

export async function POST(req: Request) {
  // ... auth + rate limit checks ...

  const stream = createUIMessageStream({
    execute: async ({ writer }) => {
      // Phase 1: Reasoning (reformulation)
      writer.write({
        type: 'data-reasoning',
        data: { status: 'reformulating', query },
        transient: true,
      });

      // Phase 2: Tool call (Exa search)
      writer.write({
        type: 'data-tool',
        id: 'exa-search',
        data: { status: 'running', query: reformulatedQuery },
      });

      const exaResults = await searchExa(reformulatedQuery);

      writer.write({
        type: 'data-tool',
        id: 'exa-search',                  // same id = reconcile (update)
        data: { status: 'completed', count: exaResults.length },
      });

      // Phase 3: Shimmer then cards
      writer.write({ type: 'data-shimmer', data: { active: true }, transient: true });

      const digestedCards = await digestForScrapbook(personName, query, exaResults);

      writer.write({ type: 'data-shimmer', data: { active: false }, transient: true });

      for (const card of digestedCards) {
        writer.write({ type: 'data-card', data: card });  // streams one at a time
      }

      // Phase 4: Sources
      writer.write({ type: 'data-sources', data: { urls: exaResults.map(r => r.url) } });

      // Text summary via streamText merged in
      const result = streamText({
        model: openrouter.chat('anthropic/claude-3.5-haiku'),
        prompt: `Summarize these findings about ${personName}...`,
      });
      writer.merge(result.toUIMessageStream());
    },
  });

  return createUIMessageStreamResponse({ stream });
}
```

Client consumption:
```typescript
// Source: https://ai-sdk.dev/docs/ai-sdk-ui/streaming-data
'use client';
import { useChat } from '@ai-sdk/react';

const { messages, sendMessage } = useChat({
  api: `/api/prospects/${prospectId}/research`,
  onData: (dataPart) => {
    if (dataPart.type === 'data-shimmer') setShimmerActive(dataPart.data.active);
    if (dataPart.type === 'data-reasoning') setReasoningState(dataPart.data);
  },
});

// Persistent data-card parts appear in message.parts
messages.flatMap(m => m.parts.filter(p => p.type === 'data-card'))
```

### Anti-Patterns to Avoid
- **Using legacy `StreamData` class:** Removed in AI SDK v6. Use `createUIMessageStream` + `writer.write()`.
- **Using the existing `chatCompletion` function from `openrouter.ts` for streaming:** It uses `fetch` and returns non-streaming response. For streaming routes, use `@openrouter/ai-sdk-provider` + `streamText`.
- **Mutating `exa-digest.ts`:** Existing enrichment pipeline uses `digestExaResults`. Create a new `research-digest.ts` that re-uses patterns but adds scrapbook fields.
- **Writing RLS policies in migration files:** Project convention (confirmed in all 3 existing migrations): enable RLS in SQL, configure policies in Supabase dashboard.
- **`import { redis } from "@/lib/redis/client"` vs `"@/lib/cache/redis"`:** Both exist. `limiters.ts` uses `@/lib/cache/redis`. Use the same import for the research rate limiter.

## Common Pitfalls

### Pitfall 1: AI SDK v5 vs v6 API Mismatch
**What goes wrong:** Using `StreamData`, `toDataStreamResponse()`, `writeData()`, `useChat({ experimental_onToolCall })` — all removed or renamed in v6.
**Why it happens:** Most tutorials and blog posts as of 2025 still reference v3/v4/v5 patterns.
**How to avoid:** The project's `ai` package is 6.0.141. Use `createUIMessageStream`, `writer.write({ type: 'data-*', data })`, `createUIMessageStreamResponse`. On client, `useChat` from `@ai-sdk/react` (separate package from `ai`).
**Warning signs:** TypeScript error `StreamData is not exported`, `toDataStreamResponse is not a function`.

### Pitfall 2: useChat vs Custom Hook Complexity
**What goes wrong:** Building a custom streaming hook when `useChat` supports custom data streams natively via `onData` callback.
**Why it happens:** The multi-phase streaming looks complex, so engineers assume they need full control.
**How to avoid:** `useChat` supports: typed data parts in `message.parts`, transient data in `onData` callback, custom API endpoint, automatic reconnect. Use it unless profiling reveals a specific limitation.
**Warning signs:** Rebuilding message state management from scratch in `useState`.

### Pitfall 3: shadcn AI Component Registry URL Wrong
**What goes wrong:** `npx shadcn@latest add message` (without URL) fails or installs wrong component.
**Why it happens:** The AI components are in a separate registry from the main shadcn UI components.
**How to avoid:** Use full URL format: `pnpm dlx shadcn@latest add https://www.shadcn.io/r/message.json`
**Warning signs:** Install succeeds but no file appears in `src/components/ui/`.

### Pitfall 4: Reconciliation ID Not Matching for Tool Phase Update
**What goes wrong:** `writer.write({ type: 'data-tool', id: 'exa-search', data: { status: 'running' } })` then later writing `{ type: 'data-tool', id: 'exa-search', data: { status: 'completed' } }` — if IDs differ, two separate parts render instead of one that updates.
**Why it happens:** The reconciliation pattern requires the exact same `id` for the same logical data part.
**How to avoid:** Assign stable IDs at the start of each streaming session. Use constants.

### Pitfall 5: `outreach_hooks` JSONB Append Race Condition
**What goes wrong:** Two concurrent pins of type `dossier_hook` both read the array, append independently, and the second write overwrites the first.
**Why it happens:** Read-modify-write pattern without DB-level atomicity.
**How to avoid:** Use PostgreSQL's `jsonb_set` or `||` operator in a single UPDATE. Pattern: `UPDATE prospects SET intelligence_dossier = jsonb_set(intelligence_dossier, '{outreach_hooks}', intelligence_dossier->'outreach_hooks' || $1::jsonb) WHERE id = $2`. Execute via admin client.

### Pitfall 6: Activity Type Enum Not Extended
**What goes wrong:** Passing `eventType: "research_scrapbook_search"` to `logProspectActivity` causes TypeScript error because `EventType` in `src/types/activity.ts` doesn't include it.
**Why it happens:** The `EventType` union type in `activity.ts` is exhaustive.
**How to avoid:** Extend `DataEventType` in `src/types/activity.ts` to add `"research_scrapbook_search"`, `"research_scrapbook_pin"` as new values. Also add to `EVENT_TITLES`. Do this in Wave 0.

### Pitfall 7: `research_messages.result_cards` vs `data-card` Stream Parts
**What goes wrong:** Confusion between what's persisted to DB and what's streamed. The streamed `data-card` parts are ephemeral; the persisted copy is in `research_messages.result_cards` JSONB.
**Why it happens:** Two representations for the same data.
**How to avoid:** The research API route must save the final card array to `research_messages.result_cards` after streaming completes (in `onFinish` or after the stream closes). The client re-renders from stream on first load; on reload it fetches from `GET sessions/{id}`.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `StreamData` + `writeData()` | `createUIMessageStream` + `writer.write()` | AI SDK v5 → v6 | Breaking change; old approach causes runtime errors on v6 |
| `import { useChat } from 'ai'` | `import { useChat } from '@ai-sdk/react'` | AI SDK v5 → v6 | useChat moved to separate package |
| `toDataStreamResponse()` | `toUIMessageStreamResponse()` / `createUIMessageStreamResponse()` | AI SDK v5 → v6 | Both exist for different scenarios |
| Batch non-streaming LLM call via `chatCompletion` | `streamText` + `@openrouter/ai-sdk-provider` | This phase introduces | Required for multi-phase streaming UX |

**Deprecated/outdated:**
- `StreamData` class: removed in AI SDK v6. Any tutorial referencing it predates the current project setup.
- `experimental_onToolCall` in useChat: replaced with typed `message.parts` in v5+.

   - What we know: `prompt-input.json` confirmed live (documentation explicitly shows URL). The `shadcn.io/ai` page confirms the component names and URL pattern.

   - What we know: Exa is ~1-3s, LLM digest is ~5-15s for 10 results, total well under 60s.
   - What's unclear: Edge case where digest runs long.
   - Recommendation: Add 30s hard timeout on the Exa + digest chain; stream partial results if timeout hit.

3. **`@ai-sdk/react` useChat `onData` type safety for custom data parts**
   - What we know: AI SDK v6 supports `useChat<MyUIMessage>` generic typing for type-safe parts.
   - What's unclear: Whether the generic typing is fully supported on `@ai-sdk/react@3.0.x` or requires newer minor.
   - Recommendation: Start with untyped `onData` callback; add generic typing if TypeScript errors appear.
