## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Application code ‚Üí Postgres RLS | Every query function uses session client (`createClient()` from `@/lib/supabase/server`) so RLS runs under the caller's JWT. No admin-client reads here. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-44-02 | Elevation of Privilege / Tampering | `updateListVisibility` / `updatePersonaVisibility` | mitigate | Functions call `supabase.from(...).update(...).eq("id",...).eq("tenant_id",...)` with the session client. RLS `UPDATE USING` clause (from plan 01) enforces creator-or-admin. If a non-creator non-admin calls this function, RLS silently updates zero rows and the function returns without error (acceptable ‚Äî the UI render-gate in plans 04-05 prevents UI exposure). No parallel JS permission check per D-09. |
| T-44-11 | Tampering via bad input | Visibility column value | mitigate | TypeScript `Visibility` union type rejects arbitrary strings at compile time. Postgres enum `visibility_mode` rejects at write time. Two layers of validation. |
| T-44-12 | Information Disclosure via query amplification | `getAllListsWithCreators` / `getAllPersonasWithCreators` | accept | These functions do NOT filter by visibility ‚Äî they trust RLS. The admin-role clause in the RLS policy yields all rows for `tenant_admin` / `super_admin`. For non-admin callers, RLS filters normally. This is intentional per D-12 + PATTERNS.md Pattern 3. Risk: if the page gate is removed, a non-admin could call these and still only see their own visible rows (safe). |
