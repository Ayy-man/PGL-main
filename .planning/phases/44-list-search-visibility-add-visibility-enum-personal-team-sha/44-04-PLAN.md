## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client-side form state → Server Action | Hidden input value is user-controllable. Server action validates via `isVisibility()` guard (plan 44-03) and RLS enforces final authorization. UI control is a convenience, not a trust boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-44-02 | Spoofing via tampered client state | Hidden `name="visibility"` input | mitigate | Server action validates input via `isVisibility()` (plan 44-03) and RLS `WITH CHECK` on INSERT enforces `created_by = auth.uid()`. User cannot spoof another user's created_by. User can set any valid `visibility` value, but that's intended — it's their own list/persona. |
| T-44-14 | Optimistic state leak | OptimisticList pre-confirm | accept | Temp row only visible to the submitting user in their own browser. No cross-user state leak possible (state is in-memory, not server-persisted). |
</threat_model>
