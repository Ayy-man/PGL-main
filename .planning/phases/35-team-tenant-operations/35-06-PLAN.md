# Plan 35-06: Non-Admin Onboarding Flow

## Goal

Create a simplified onboarding page for agent/assistant users. Currently all invited users land on `/onboarding/confirm-tenant` which shows org configuration (name, slug, theme, logo) that non-admin users shouldn't see or configure. The middleware already references `/onboarding/set-password` in its allow-list but the page doesn't exist.

## Tasks

### Task 1: Create set-password onboarding page
- Create `src/app/onboarding/set-password/page.tsx`
- Simple form: welcome message with org name, password input, confirm password input, submit button
- On submit: call `completeUserOnboarding` from `src/app/actions/onboarding.ts` (already exists)
- After success: redirect to `/{orgId}` dashboard
- Match existing onboarding page styling

### Task 2: Update auth callback routing
- Edit `src/app/api/auth/callback/route.ts`
- After token exchange, check user's role from `app_metadata`
- Route based on role:
  - `tenant_admin` with `onboarding_completed=false` -> `/onboarding/confirm-tenant`
  - `agent`/`assistant` with `onboarding_completed=false` -> `/onboarding/set-password`
  - All other cases: existing routing logic

### Task 3: Update middleware onboarding redirect
- Edit `src/middleware.ts`
- In the onboarding redirect logic, check role:
  - Non-admin users (`agent`/`assistant`) -> redirect to `/onboarding/set-password`
  - Admin users (`tenant_admin`) -> redirect to `/onboarding/confirm-tenant` (existing behavior)
- Ensure `/onboarding/set-password` is in the onboarding allow-list (it may already be)

## Verification
- [ ] Invited agent clicks email link -> lands on set-password page (not confirm-tenant)
- [ ] Set-password page shows welcome + org name + password form only
- [ ] After setting password, user redirected to dashboard
- [ ] Tenant admin still sees full onboarding form
- [ ] Middleware correctly routes by role
