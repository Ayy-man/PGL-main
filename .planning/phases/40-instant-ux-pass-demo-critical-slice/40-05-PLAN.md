```tsx
const handleConfirmDelete = async () => {
  if (!pendingDeleteList) return;
  setDeleting(true);

  const previousLists = lists;
  const deletedId = pendingDeleteList.id;
  const deletedName = pendingDeleteList.name;
  setLists(prev => prev.filter(l => l.id !== deletedId));

  const restoreList = (prev: List[]) => {
    return prev.some(l => l.id === deletedId) ? prev : previousLists;
  };

  toast({
    title: `Deleted "${deletedName}"`,
    action: (
      <ToastAction altText="Undo" onClick={() => setLists((prev) => restoreList(prev))}>
        Undo
      </ToastAction>
    ),
  });

  const result = await deleteListAction(deletedId);
  if (!result.success) {
    setLists(previousLists);
    toast({ title: "Delete failed", description: result.error || "Could not delete list.", variant: "destructive" });
  }

  setDeleting(false);
  setPendingDeleteList(null);
};
```

Import `ToastAction` from `@/components/ui/toast`. Import `List` type if not already.

Note: Undo click does NOT re-call any server action because the server delete has not yet fired when the toast appears IF the user clicks Undo within the server action latency window. But the server action DID fire right after the setLists. So clicking Undo after the server has already deleted will restore the row in UI only, and it'll flicker out on the next `revalidatePath` / `router.refresh`. Acceptable for v1 — the Undo is a *visual* undo that works cleanly when the server hasn't yet committed. If the user wants real server-side undo, that's a separate feature (needs an undelete endpoint). Document this behavior with a code comment inline.
